# Audit Summary (Lab 3: Lemma & POS)

## Загальна статистика
- **Інструмент:** Stanza (uk)
- **Згенеровані фічі:** `lemma_text` (леми), `pos_seq` (POS-теги).
- **Модель:** TF-IDF + Logistic Regression (Класифікація текстів, 3 класи).

## Результати A/B/C тесту (Baseline)
1. **Baseline 1 (RAW `processed_v2`):**
   - Accuracy: **0.4225**
   - Macro-F1: **0.4225**
2. **Baseline 2 (LEMMA `text_lemma`):**
   - Accuracy: **0.3450** - Macro-F1: **0.3447**
3. **Baseline 3 (POS tags only `text_pos`):**
   - Accuracy: **0.3375** - Macro-F1: **0.3354**

## Error Analysis та Висновок
Під час аналізу крайових випадків (`tests/ling_edge_cases.jsonl`) було виявлено, що датасет містить велику кількість артефактів машинного перекладу (неіснуючі слова, як-от "краткочісний", "ладенка", "капуті").

**Чому результати погіршилися?**
1. **Лематизація (34.50%):** Зведення слів до називних відмінків та інфінітивів усунуло морфологічні зв'язки, які допомагали розрізняти суперечності та логічні наслідки. Для TF-IDF моделі втрата закінчень виявилася критичною.
2. **POS-теги (33.75%):** Модель, навчена виключно на синтаксисі (n-грамах POS-тегів), показала результат на рівні випадкового вгадування (для 3 класів це ~33.3%). Причина: для задачі NLI семантика (значення конкретних слів) набагато важливіша за синтаксичну структуру (знання про те, що речення складається з `NOUN VERB`, не допомагає зрозуміти, чи суперечать ці дії одна одній).

**Рішення для проєкту:**
Леми та чисті POS-теги **не беремо** для навчання базових ML-моделей. Звичайний очищений текст (`processed_v2`) зберігає набагато більше контекстної інформації, що доводить найвищий результат Accuracy (42.25%). Лінгвістичні фічі залишаємо в датасеті лише як можливі мета-дані для складніших архітектур у майбутньому.
